当前主要观察

业务规则

  • 球员成熟后进入“挖人窗口”，窗口内可被截获部分积分。
  • 窗口结束后才允许 `harvest()` 正式结算。
  • `placeScarecrow()` 已按品牌语义包装为“防挖保护”。

安全边界

  • 用户业务入口不再自动触发运营池提现，降低外部调用复杂度。
  • `flushOpsVault()` 仅允许 `owner` 或 `guardian` 主动触发。
  • 工厂仍通过 `VaultPortal` 入口控制部署路径。

本轮发现

1. 赛季滚动上限可能导致过期赛季继续记分

`_rolloverIfNeeded()` 最多只补 20 轮；若长时间无人交互，函数在未追平到当前时间时也会继续执行后续业务逻辑，新的积分可能被记入理论上已结束的赛季。

结论：这是正式审计中的主要阻断项，建议在未追平时直接 revert，或改为显式批量结转流程。

2. 最低持仓门槛会阻断收割

`harvest()` 也要求地址持有至少 100 万代币，若用户参赛后余额跌破门槛，将无法完成收割，已投入的球员与时间成本会被卡住。

3. UI schema 使用未定义的 `duration` 类型

`playerConfig()` 在 schema 输出里使用 `duration`，但底层 schema 规范仅定义了 `time` 与标准 ABI 类型，通用前端可能无法正确渲染。

4. 部署仍依赖硬编码地址假设

`flap` 基座中的 `Portal / Guardian` 地址按链 ID 写死，若部署链、代理地址或权限假设不一致，会直接影响工厂权限与运维边界。

设计说明

  • Token 精度假设： `SoccerVault` 仅支持 `18 decimals` 的 `taxToken`，构造函数已强校验；所有门槛、签约成本、挖人成本与积分精度均以此为前提。
  • 受控暂停机制： 当 backlog 赛季超过单次自动追平上限时，业务入口会因 `RolloverPending()` 暂停；此时需调用 `syncSeasons(maxLoops)` 手动追平赛季后再恢复使用。
  • 部署链假设： `Portal / Guardian` 地址按链 ID 硬编码，仅适配对应 BNB 主网/测试网环境，部署与审计均应基于该假设。
  • 审计 scope： 除 `SoccerVault.sol` 与 `SoccerVaultFactory.sol` 外，建议把整个 `contracts\flap` 目录作为上下文依赖一并提交。
  • 前端与运维要求： 前端应识别 `RolloverPending()` 并提示执行 `syncSeasons()`；监听脚本需使用当前品牌化后的事件名。

建议下一步

  • 修复 `_rolloverIfNeeded()`：当超过 `MAX_ROLLOVER_LOOPS` 仍未追平当前时间时，不应继续执行业务记分。
  • 放宽 `harvest()` 的最低持仓限制，至少允许已参赛阵位完成收割与结算。
  • 把 `playerConfig()` schema 中的 `duration` 改为 `uint256` 或 `time` 兼容表示，并更新前端格式化逻辑。
  • 若准备送第三方审计，附带整个 `contracts\flap` 目录作为上下文依赖，并在 scope 说明里写明 `Portal / Guardian` 为硬编码地址假设。
